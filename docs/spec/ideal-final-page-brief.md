# Идеальный финальный SEO-бриф страницы

Этот документ описывает целевой результат, который мы хотим получать после анализа запроса: финальный ТЗ-бриф для создания страницы. Он не привязан к текущей реализации pipeline. Сначала фиксируем, каким должен быть качественный бриф, и только потом под этот результат настраиваем сбор данных, анализ, генерацию, проверки и scoring.

## Главный принцип

Финальный бриф - это не SEO-отчет, не выгрузка конкурентов, не список ключей, не outline и не черновик статьи. Это рабочее задание на страницу.

После чтения брифа должно быть понятно:

- какую страницу нужно создать;
- зачем она нужна пользователю;
- какой интент она закрывает;
- какие запросы входят в страницу, а какие нет;
- что именно должен написать копирайтер;
- какие блоки и элементы нужны на странице;
- какие факты нужно подтвердить источниками;
- чем страница должна быть лучше конкурентов;
- как принять готовый текст и готовую страницу.

Идеальный бриф должен быть пригоден для любого запроса, но не должен превращать все запросы в один шаблон статьи. Формат документа универсальный, а структура страницы и содержание всегда зависят от анализа.

## Что не считается финальным брифом

Плохой результат:

- "напишите статью на тему";
- список H2 без задач разделов;
- таблица конкурентов без выводов;
- набор ключей без кластеризации;
- пересказ SERP;
- HTML-макет без контента;
- SEO-отчет, из которого нельзя понять, что писать;
- статья, написанная вместо ТЗ;
- общий чек-лист без решений по конкретной странице.

Хороший результат:

- объясняет, какую страницу делаем;
- показывает, почему именно такую;
- переводит анализ в конкретные требования к тексту, структуре и UX;
- отделяет обязательный базовый уровень от возможностей сделать лучше;
- дает критерии приемки.

## Роли, для которых пишется бриф

### Копирайтер

Должен понять:

- кто пользователь;
- какая у него задача;
- что писать в каждом разделе;
- какие вопросы закрыть;
- какие примеры, таблицы, сценарии и формулировки нужны;
- где нельзя фантазировать;
- какие источники использовать.

### Редактор

Должен понять:

- достаточно ли текста для решения задачи;
- нет ли воды, повторов и шаблонных вступлений;
- закрыты ли ключевые вопросы;
- не смешаны ли разные интенты;
- проверены ли факты;
- соответствует ли тон теме и рискам.

### SEO-специалист

Должен понять:

- есть ли соответствие интенту;
- не будет ли каннибализации;
- покрыт ли кластер;
- корректны ли title, H1, description;
- нужны ли schema, FAQ, внутренние ссылки;
- чем страница конкурирует в выдаче.

### Дизайнер или разработчик

Должен понять:

- какие блоки нужны на странице;
- что должно быть на первом экране;
- где нужны таблицы, карточки, формы, калькуляторы, чек-листы;
- как пользователь должен двигаться по странице;
- какие элементы доверия и навигации нужны.

## Общая структура финального брифа

Финальный документ должен состоять из следующих разделов.

1. Короткое решение по странице.
2. Паспорт страницы.
3. Интент и пользовательская задача.
4. Кластер запросов и границы страницы.
5. Сущности, обязательные факты и SEO-триплеты.
6. Конкурентная картина.
7. Стратегия нашей страницы.
8. Title, H1, description и первый экран.
9. Подробный план контента для копирайтера.
10. UX, визуальная структура и функциональные блоки.
11. Источники, факты и экспертная проверка.
12. Внутренние ссылки и следующий шаг пользователя.
13. Schema и технические SEO-требования.
14. Редакционные правила.
15. Чек-лист приемки.

Ниже описано, что должен содержать каждый раздел.

## 1. Короткое решение по странице

Это первый блок брифа. Он нужен, чтобы за 30 секунд понять суть.

Обязательные поля:

- основной запрос;
- тип страницы;
- решение: создать новую, обновить существующую, объединить, разделить или не делать;
- главная задача страницы;
- главный риск;
- главный способ выиграть у конкурентов;
- статус готовности: `ready_for_copywriter`, `ready_for_design_and_build`, `requires_expert_review`, `blocked`, `draft`.

Формат:

```markdown
## Короткое решение

Делаем: <тип страницы>
Почему: <логика интента и SERP>
Для кого: <аудитория>
Главная задача: <что пользователь должен решить>
Главное отличие: <чем будем лучше среднего конкурента>
Риск: <что может испортить страницу>
Статус: <...>
```

## 2. Паспорт страницы

Паспорт фиксирует базовые параметры.

Нужно указать:

- основной запрос;
- язык и регион;
- дата анализа;
- тип страницы;
- этап воронки;
- аудитория;
- уровень знаний пользователя;
- YMYL-риск;
- нужна ли экспертная проверка;
- существующие страницы, с которыми возможна каннибализация;
- целевое действие пользователя.

Паспорт не должен превращаться в длинное рассуждение. Это справочный блок.

## 3. Интент и пользовательская задача

Здесь нужно ответить не "о чем запрос", а "что человек пытается сделать".

Обязательные вопросы:

- Что пользователь хочет получить сразу?
- Какую проблему он решает?
- Какой результат он считает успешным?
- Что он уже, вероятно, знает?
- Какие страхи, сомнения или ограничения у него есть?
- Насколько запрос информационный, коммерческий, транзакционный, навигационный или смешанный?
- Какие интенты нельзя смешивать на этой странице?

Пример хорошей формулировки:

```markdown
Пользователь не просто ищет определение. Ему нужен практический порядок действий, развилка по ситуациям, список документов, ограничения, сроки и понимание рисков. Поэтому страница должна начинаться с короткого ответа и сценариев, а не с общего вступления.
```

## 4. Кластер запросов и границы страницы

Кластер нужен не для вставки ключей, а для проектирования страницы.

Группы кластера:

- **Core** - запросы, которые страница обязана закрыть напрямую.
- **Supporting** - вопросы, которые раскрываются внутри разделов.
- **Adjacent** - близкие темы, которые можно кратко упомянуть или связать внутренней ссылкой.
- **Excluded** - запросы, которые нельзя покрывать здесь, потому что у них другой интент.

Правила:

- дубли и морфологические варианты объединяются;
- разные формулировки одного смысла не раздувают outline;
- каждый supporting-запрос должен быть привязан к конкретному разделу;
- excluded-запросы нужны обязательно, чтобы не получить размытую страницу;
- если нет внешних сервисов частотности, нельзя писать про "высокочастотный" или "низкочастотный" без данных.

Формат:

```markdown
### Core
- <запрос> -> закрываем в первом экране и разделе <...>

### Supporting
- <вопрос> -> раздел <...>

### Adjacent
- <тема> -> коротко упомянуть / дать ссылку / вынести отдельно

### Excluded
- <запрос> -> не покрываем, потому что <причина>
```

## 5. Сущности, обязательные факты и SEO-триплеты

Этот раздел нужен, чтобы страница не была рерайтом конкурентов. Он описывает предметную область: какие сущности есть в теме, какие факты пользователь обязан получить, какие связи между ними нужно объяснить и чем можно обогатить страницу сверх пересказа SERP.

Коротко:

- **сущность** - главный объект темы: услуга, процедура, инструмент, болезнь, товар, закон, показатель, место, событие, человек, организация;
- **обязательный факт** - информация, без которой ответ будет неполным или опасным;
- **SEO-триплет** - смысловая единица вида `сущность -> свойство/отношение -> значение`, иногда с условиями, датой, источником и ограничениями.

Пример:

```markdown
Сущность: развод при наличии детей
Триплет: развод при наличии несовершеннолетних детей -> обычно проходит через -> суд
Условие: если нет исключений, предусмотренных законом
Источник: официальный правовой источник
Где раскрыть: первый экран, раздел "Куда обращаться"
Риск: нельзя обещать универсальный порядок без учета ситуации
```

SEO-триплеты не заменяют текст и не должны механически вставляться в статью. Это карта смысла и фактов, по которой проверяется полнота страницы.

### 5.1. Что анализировать на уровне сущности

Нужно определить:

- главную сущность страницы;
- связанные сущности;
- свойства главной сущности;
- условия и исключения;
- этапы процесса;
- участников процесса;
- документы, инструменты, показатели, сроки, цены или ограничения;
- типичные ошибки;
- риски для пользователя;
- источники истины;
- точки, где данные могут устареть.

Для разных типов запросов сущности будут разными.

Примеры:

```markdown
Юридическая тема:
сущности: процедура, суд, заявление, дети, документы, сроки, госпошлина, соглашение, алименты.

Медицинская тема:
сущности: состояние, симптомы, диагностика, лечение, противопоказания, врач, срочность, источники.

Калькулятор:
сущности: показатель, формула, входные данные, единицы измерения, ограничения, интерпретация результата.

Товарная страница:
сущности: продукт, характеристики, совместимость, материалы, сценарии использования, гарантии, сравнения.
```

### 5.2. Обязательные факты

В брифе нужно отдельно указать факты, которые должны быть в тексте или интерфейсе.

Обязательный факт отвечает хотя бы одному критерию:

- без него пользователь не может действовать;
- без него ответ будет вводить в заблуждение;
- его ожидает выдача и пользователь;
- он влияет на безопасность, деньги, здоровье, юридический результат или решение о покупке;
- он часто путается в конкурентных материалах;
- он является основанием для выбора сценария.

Формат:

```markdown
| Факт | Почему обязателен | Где раскрыть | Источник | Условия/исключения | Риск ошибки |
|---|---|---|---|---|---|
```

### 5.3. SEO-триплеты

Триплеты нужны, чтобы разложить тему на проверяемые смысловые утверждения.

Базовый формат:

```markdown
<сущность> -> <свойство или отношение> -> <значение>
```

Расширенный формат:

```markdown
Сущность:
Отношение:
Значение:
Условие:
Источник:
Дата актуальности:
Где раскрыть:
Тип: baseline | enrichment | risk | excluded
```

Типы триплетов:

- **baseline** - обязательный минимум, без которого страница неполная;
- **enrichment** - обогащение, которое делает страницу полезнее конкурентов;
- **risk** - утверждение, где нужна осторожность, источник или экспертная проверка;
- **excluded** - смысл, который относится к другой странице и не должен размывать интент.

### 5.4. Как использовать триплеты в брифе

Триплеты должны попадать не отдельной сырой таблицей, а в решения:

- в первый экран, если это главный ответ;
- в outline, если это обязательный раздел;
- в таблицу фактов, если нужна проверка источником;
- в UX-блок, если связь проще показать таблицей, схемой или шагами;
- в FAQ, если это реальный следующий вопрос пользователя;
- в excluded-темы, если смысл нельзя покрывать на этой странице;
- в schema, если триплет поддерживает видимую структурированную информацию.

Пример:

```markdown
Триплет: процедура -> имеет этапы -> подача заявления, рассмотрение, решение, регистрация
Решение в брифе: нужен блок "Порядок действий" в формате пошаговой схемы.
Критерий приемки: пользователь понимает последовательность без чтения всей статьи.
```

### 5.5. Как это помогает не делать рерайт

Конкуренты показывают, что уже есть в выдаче. Сущности и триплеты показывают, что тема реально требует раскрыть.

Поэтому бриф должен сравнивать:

- что обязано быть по сущности темы;
- что уже есть у конкурентов;
- чего у конкурентов нет;
- что можно объяснить понятнее;
- какие связи лучше показать структурно;
- какие факты нужно проверить у первоисточников;
- какие утверждения нельзя брать из конкурентных статей.

Итоговый вывод:

```markdown
Не "у конкурентов есть раздел X, добавим раздел X", а "пользователю нужно понять связь A -> B -> C, конкуренты раскрывают ее слабо, поэтому мы делаем таблицу/схему/пример".
```

### 5.6. Что должно попасть в финальный бриф

Обязательный блок:

```markdown
## Сущности, факты и SEO-триплеты

Главная сущность:
Связанные сущности:

### Обязательные факты
| Факт | Почему обязателен | Где раскрыть | Источник | Риск |
|---|---|---|---|---|

### SEO-триплеты
| Сущность | Отношение | Значение | Тип | Где использовать | Проверка |
|---|---|---|---|---|---|

### Обогащение
- <какую связь/пример/таблицу/сценарий добавить, чтобы страница была не рерайтом>

### Не включать
- <сущность или связь> -> <почему это другой интент>
```

## 6. Конкурентная картина

Это один из ключевых разделов. Его цель - не собрать "что есть у конкурентов", а понять, какие решения должна принять наша страница.

Конкурентный анализ должен отвечать на вопрос:

> Что выдача и сильные страницы показывают нам о том, какой контент, структура, доверие и UX нужны пользователю?

### 6.1. Кого считать конкурентами

Нужно разделять три типа конкурентов.

**Поисковые конкуренты** - страницы, которые реально стоят в SERP по запросу. Они показывают, что поисковая система считает релевантным сейчас.

**Контентные конкуренты** - страницы, которые лучше всего закрывают пользовательскую задачу, даже если они не в топе по основному запросу. Их можно найти по supporting-запросам.

**Бизнес-конкуренты** - сайты, с которыми проект реально конкурирует за заявку, продажу, доверие или внимание. Они могут не совпадать с поисковыми конкурентами.

В финальный бриф обычно попадают поисковые и контентные конкуренты. Бизнес-конкурентов нужно учитывать, если страница коммерческая или влияет на конверсию.

### 6.2. Сколько конкурентов анализировать

Минимум:

- топ-5 органических результатов по основному запросу;
- 2-3 результата по важным supporting-запросам;
- 1-2 сильных страницы вне топа, если они явно лучше закрывают задачу.

Для сложных YMYL-тем:

- обязательно добавить официальные источники;
- отдельно отметить, какие конкуренты опираются на источники, а какие пересказывают друг друга;
- не считать непроверенные коммерческие страницы источником истины.

Не нужно анализировать все подряд. Лучше 8 хорошо разобранных страниц, чем 30 URL в таблице без выводов.

### 6.3. Что фиксировать по каждому конкуренту

По каждому важному конкуренту нужно собрать не только H2.

Минимальная карточка конкурента:

```markdown
URL:
Тип сайта:
Тип страницы:
Позиция/источник:
Title:
H1:
Snippet:
Дата обновления:
Автор/эксперт:
Что видно на первом экране:
Главное обещание страницы:
Какие интенты закрывает:
Какие интенты смешивает:
Сильные блоки:
Слабые места:
Источники и доказательства:
UX-решения:
CTA/конверсия:
Что можно использовать как сигнал:
Что нельзя копировать:
```

Важно: карточка конкурента - это сырье. В финальный бриф она попадает только в сжатом виде, если из нее есть решение для нашей страницы.

### 6.4. Что именно анализировать у конкурентов

#### Тип страницы

Нужно понять, какой формат доминирует:

- инструкция;
- статья-объяснение;
- юридический/медицинский/финансовый справочник;
- коммерческая услуга;
- каталог;
- сравнение;
- калькулятор;
- FAQ;
- лендинг;
- агрегатор;
- официальный документ;
- смешанный формат.

Вывод для брифа:

- какой тип страницы делаем мы;
- можно ли выиграть другим форматом;
- нельзя ли делать статью, если выдача требует инструмента, услуги или официального ответа.

#### Первый экран

Нужно смотреть:

- дает ли конкурент ответ сразу;
- есть ли навигация по сценариям;
- есть ли дата обновления;
- виден ли автор или эксперт;
- есть ли CTA;
- не закрывает ли контент рекламный блок;
- какое обещание страницы пользователь видит до скролла.

Вывод для брифа:

- что обязательно должно быть на нашем первом экране;
- как быстро нужно дать ответ;
- нужен ли блок "коротко";
- нужна ли развилка по ситуациям;
- какие элементы доверия показать сразу.

#### Структура и порядок разделов

Нужно смотреть:

- с чего начинают сильные страницы;
- какие разделы повторяются у большинства;
- какие разделы есть только у лучших;
- где конкуренты уходят в лишнюю теорию;
- какие вопросы стоят слишком низко, хотя важны пользователю;
- какие темы смешаны неправильно.

Вывод для брифа:

- какие разделы являются baseline;
- какие разделы можно убрать;
- какие разделы нужно поднять выше;
- какой порядок лучше для пользователя.

#### Глубина раскрытия

Нужно оценить:

- дают ли конкуренты только общие ответы или практические шаги;
- есть ли условия, исключения, варианты;
- объясняют ли они "что делать, если";
- есть ли примеры;
- есть ли таблицы, чек-листы, документы, формы, расчеты;
- хватает ли информации, чтобы пользователь мог действовать.

Вывод для брифа:

- какой уровень глубины нужен;
- где обязательны примеры;
- где нужна таблица или чек-лист;
- какие сценарии нужно раскрыть отдельно.

#### Факты, источники и доверие

Нужно смотреть:

- есть ли ссылки на официальные источники;
- указаны ли даты обновления;
- кто автор;
- есть ли экспертная проверка;
- не устарели ли факты;
- есть ли юридические, медицинские, финансовые или регуляторные утверждения;
- не повторяют ли конкуренты сомнительные данные друг за другом.

Вывод для брифа:

- какие факты нужно подтвердить;
- где нужна экспертная вычитка;
- какие источники обязательны;
- какие утверждения нельзя писать без проверки;
- нужна ли дата актуальности в видимой части страницы.

#### Полезные форматы

Нужно смотреть, какие форматы реально помогают:

- таблицы сравнения;
- пошаговые инструкции;
- чек-листы;
- timeline;
- калькуляторы;
- формы;
- примеры документов;
- сценарные блоки;
- FAQ;
- карточки вариантов;
- предупреждения;
- списки ошибок.

Вывод для брифа:

- какие форматы обязательны;
- какие форматы дадут преимущество;
- какие форматы не нужны и будут шумом.

#### Покрытие вопросов

Нужно смотреть:

- какие вопросы повторяются у конкурентов;
- какие вопросы есть в подсказках/сниппетах/related queries;
- какие вопросы конкуренты раскрывают поверхностно;
- какие вопросы пользователь должен задать дальше;
- какие вопросы относятся к другой странице.

Вывод для брифа:

- какие вопросы включить в основной текст;
- какие вынести в FAQ;
- какие закрыть внутренними ссылками;
- какие исключить.

#### UX и читаемость

Нужно смотреть:

- легко ли сканировать страницу;
- есть ли якорная навигация;
- не слишком ли длинные абзацы;
- удобно ли читать на мобильном;
- можно ли быстро найти ответ;
- помогает ли дизайн понять материал;
- не мешают ли баннеры, попапы, реклама, тяжелые блоки.

Вывод для брифа:

- нужна ли sticky-навигация;
- нужны ли компактные summary-блоки;
- как оформить таблицы на мобильном;
- где нужны визуальные акценты;
- чего избегать в дизайне.

#### Конверсия и следующий шаг

Для коммерческих и сервисных страниц нужно смотреть:

- где стоит CTA;
- не конфликтует ли CTA с информационным интентом;
- есть ли доверие перед предложением услуги;
- какой следующий шаг предлагает конкурент;
- есть ли формы, консультация, расчет, запись, скачивание;
- насколько агрессивно продает страница.

Вывод для брифа:

- нужен ли CTA;
- где он должен стоять;
- какой CTA уместен;
- нужно ли сначала дать полезный ответ, а потом предлагать действие;
- какие внутренние ссылки поддерживают путь пользователя.

#### Schema и SERP features

Нужно смотреть:

- есть ли FAQ на странице и в выдаче;
- есть ли HowTo, Article, Breadcrumb, Product, Review, LocalBusiness или другой релевантный тип;
- какие расширенные сниппеты показывает выдача;
- соответствует ли schema видимому контенту.

Вывод для брифа:

- нужна ли schema;
- какие видимые блоки должны поддерживать schema;
- нельзя ли использовать разметку, если контента на странице нет.

### 6.5. Как превращать наблюдения в решения

Каждый конкурентный сигнал должен проходить цепочку:

```markdown
Сигнал -> Значение -> Решение для нашей страницы -> Где в ТЗ -> Критерий приемки -> Что не копировать
```

Пример:

```markdown
Сигнал: 4 из 5 конкурентов дают список документов, но прячут его после длинного вступления.
Значение: список документов является обязательным ожиданием, но текущая выдача неудобна.
Решение: дать короткий список документов в первом экране и подробную таблицу ниже.
Где в ТЗ: первый экран, раздел "Документы", UX-блок "чек-лист".
Критерий приемки: пользователь видит базовый список до скролла или сразу после первого абзаца.
Что не копировать: длинное вступление перед практической информацией.
```

Если сигнал не приводит к решению, его не нужно включать в финальный бриф.

### 6.6. Baseline и преимущество

Нужно разделять два уровня.

**Baseline** - то, без чего страница не будет соответствовать выдаче и ожиданиям пользователя. Например: определение, условия, документы, сроки, цена, порядок действий, FAQ, дата актуальности.

**Преимущество** - то, что может сделать страницу лучше конкурентов. Например: сценарная развилка, таблица вариантов, понятный первый экран, более свежие источники, экспертная проверка, интерактивный инструмент, сильные примеры.

В брифе нельзя продавать baseline как уникальное преимущество. Если все конкуренты дают список документов, наличие списка документов - это минимум, а не отличие.

Формат:

```markdown
Baseline:
- <обязательный элемент> -> почему нужен

Преимущество:
- <что делаем лучше> -> какую проблему конкурентов решает
```

### 6.7. Типовые ошибки конкурентного анализа

Нельзя:

- копировать структуру лидера без понимания интента;
- считать частоту повторения H2 доказательством важности;
- переносить все разделы конкурентов в нашу страницу;
- игнорировать первый экран;
- смешивать поисковых и бизнес-конкурентов без пометки;
- делать выводы по одному конкуренту;
- считать коммерческие статьи источником фактов;
- включать в ТЗ слабые блоки только потому, что они есть у всех;
- путать "конкуренты делают" и "пользователю нужно";
- не фиксировать, что именно мы делаем иначе.

### 6.8. Что должно попасть в финальный бриф

В итоговый бриф не нужно вставлять огромную таблицу всех конкурентов. Нужно вставить сжатые решения.

Обязательные элементы:

```markdown
## Конкурентная картина

### Доминирующий тип выдачи
<какие страницы ранжируются и что это значит>

### Baseline-ожидания
- <элемент> -> <почему обязателен>

### Пробелы конкурентов
- <проблема> -> <как используем>

### Решения для нашей страницы
| Сигнал | Что значит | Решение | Где в ТЗ | Приемка |
|---|---|---|---|---|

### Не копировать
- <что у конкурентов слабое и почему не берем>
```

## 7. Стратегия нашей страницы

После интента, кластера и конкурентов нужно сформулировать стратегию.

Обязательные поля:

- какую страницу делаем;
- какой формат выбираем;
- что будет главным ответом;
- как пользователь будет двигаться по странице;
- что является обязательным минимумом;
- где будет наше преимущество;
- что не включаем;
- какие риски нужно контролировать.

Формат:

```markdown
Стратегия:
Мы делаем <тип страницы>, которая сначала дает <быстрый ответ>, затем помогает выбрать <сценарий>, потом раскрывает <важные детали> и завершает <следующим шагом>. Страница выигрывает не объемом текста, а <конкретное преимущество>.
```

## 8. Title, H1, description и первый экран

Бриф должен дать не просто варианты мета-тегов, а требования к первому контакту пользователя со страницей.

Нужно указать:

- основной `title`;
- альтернативный `title`;
- `H1`;
- `meta description`;
- первый абзац;
- блок "коротко", если нужен;
- сценарные кнопки или якоря, если нужны;
- дата обновления, автор, эксперт или источник доверия;
- первый CTA, если он уместен.

Требования:

- title не должен обещать больше, чем есть на странице;
- H1 должен быть понятен человеку, а не только поисковой системе;
- первый экран должен подтвердить интент;
- для сложных тем нужно сразу показать актуальность и доверие.

## 9. Подробный план контента для копирайтера

Это центральная часть брифа.

Для каждого раздела нужно указать:

- заголовок;
- задача раздела;
- вопрос пользователя;
- что именно написать;
- какие факты нужны;
- какие источники использовать;
- какой формат блока нужен;
- какие запросы покрываются;
- чего не писать;
- критерий готовности.

Шаблон раздела:

```markdown
### H2: <заголовок>

Задача:
Вопрос пользователя:
Что написать:
Обязательные факты:
Источники:
Формат:
Покрываемые запросы:
Не писать:
Критерий готовности:
```

Раздел нельзя включать, если у него нет задачи. Outline должен быть картой смысла, а не списком заголовков.

## 10. UX, визуальная структура и функциональные блоки

Страница должна быть не только текстом. Бриф должен описывать, как пользователь будет воспринимать информацию.

Нужно указать:

- композицию первого экрана;
- порядок основных блоков;
- нужна ли якорная навигация;
- где нужны таблицы;
- где нужны чек-листы;
- где нужны карточки сценариев;
- где нужны предупреждения;
- где нужны интерактивные элементы;
- как таблицы и блоки должны работать на мобильном;
- какие элементы доверия должны быть видны.

Принцип:

```markdown
Если пользователь должен сравнить варианты - нужна таблица или карточки.
Если пользователь должен выполнить процесс - нужны шаги.
Если пользователь должен проверить готовность - нужен чек-лист.
Если пользователь должен выбрать сценарий - нужна развилка.
Если пользователь должен рассчитать результат - нужен калькулятор.
```

## 11. Источники, факты и экспертная проверка

Этот раздел обязателен для тем, где ошибка может навредить пользователю: право, медицина, финансы, налоги, госуслуги, безопасность, здоровье, работа с деньгами.

Нужно указать:

- какие факты требуют подтверждения;
- какие источники являются приоритетными;
- какие данные могут устареть;
- что нужно перепроверить перед публикацией;
- где нужна экспертная вычитка;
- какие утверждения запрещены;
- какие формулировки должны быть осторожными.

Формат:

```markdown
| Факт | Источник | Где используется | Проверить перед публикацией | Риск |
|---|---|---|---|---|
```

Нельзя:

- придумывать сроки, ставки, цены, законы, документы;
- ссылаться на конкурента как на первоисточник;
- писать "гарантированно", если результата нельзя гарантировать;
- скрывать дату актуальности в темах, где она важна.

## 12. Внутренние ссылки и следующий шаг пользователя

Бриф должен проектировать путь пользователя после ответа.

Нужно указать:

- какие темы вынести на отдельные страницы;
- куда ссылаться внутри текста;
- какой следующий шаг логичен;
- где уместен CTA;
- какие ссылки помогают пользователю, а не только SEO.

Если запрос информационный, CTA должен быть аккуратным. Если запрос коммерческий, полезный ответ все равно не должен исчезать за продажей.

## 13. Schema и технические SEO-требования

Schema добавляется только под видимый контент.

Нужно указать:

- рекомендуемый тип разметки;
- какие видимые блоки ее поддерживают;
- canonical;
- index/noindex;
- breadcrumbs;
- требования к H-структуре;
- требования к датам;
- изображения, если они важны;
- внутренние ссылки;
- технические ограничения.

Нельзя добавлять FAQ-разметку, если FAQ нет на странице. Нельзя размечать инструкцию как HowTo, если на странице нет реальных шагов.

## 14. Редакционные правила

Бриф должен задавать качество письма.

Писать:

- прямо;
- конкретно;
- с практической пользой;
- с примерами там, где без них непонятно;
- с объяснением терминов;
- с отделением фактов от рекомендаций;
- в тоне, подходящем теме.

Не писать:

- длинные вступления;
- SEO-воду;
- повторы одного смысла;
- общие фразы без решения;
- искусственные ключевые фразы;
- неподтвержденные утверждения;
- кликбейт;
- имитацию экспертности.

## 15. Чек-лист приемки

Финальный бриф должен содержать checklist, по которому можно принять контент и страницу.

Минимальный чек-лист:

- [ ] Страница отвечает на основной запрос в первом экране или сразу после него.
- [ ] Тип страницы соответствует интенту.
- [ ] Core-запросы закрыты напрямую.
- [ ] Supporting-запросы распределены по разделам.
- [ ] Adjacent и excluded-темы не смешаны с основной страницей.
- [ ] Главная сущность, связанные сущности и обязательные факты определены.
- [ ] SEO-триплеты распределены по baseline, enrichment, risk и excluded.
- [ ] Конкурентный анализ превращен в решения, а не оставлен таблицей наблюдений.
- [ ] Baseline и преимущества разделены.
- [ ] Каждый раздел outline имеет задачу и критерий готовности.
- [ ] Факты и источники сопоставлены с разделами.
- [ ] Для YMYL-тем указана экспертная проверка.
- [ ] Первый экран описан как UX-блок, а не только как текст.
- [ ] Таблицы, чек-листы, формы и интерактивы указаны там, где они реально нужны.
- [ ] Внутренние ссылки помогают пользовательскому пути.
- [ ] Schema соответствует видимому содержанию.
- [ ] Копирайтер может написать страницу без дополнительного устного объяснения.
- [ ] Разработчик или дизайнер понимает, какие блоки нужны.
- [ ] Редактор понимает, как принимать результат.

```markdown
- [ ] Конкурентные сигналы превращены в решения.
- [ ] Каждый раздел имеет задачу.
- [ ] Факты подтверждены.
- [ ] UX-блоки описаны.
- [ ] Текст можно писать без дополнительных пояснений.
```

## Как этот документ должен влиять на pipeline

Pipeline нужно строить не вокруг того, что легко собрать, а вокруг того, что требуется финальному брифу.

Значит, будущий workflow должен уметь:

1. Определить интент и тип страницы.
2. Собрать и нормализовать кластер.
3. Выбрать релевантных конкурентов.
4. Разобрать конкурентов не только по H2, а по интенту, первому экрану, блокам, глубине, доверию, UX и пробелам.
5. Превратить конкурентные сигналы в решения для нашей страницы.
6. Разделить baseline и преимущества.
7. Собрать section-by-section ТЗ для копирайтера.
8. Добавить UX и визуальные требования.
9. Сопоставить факты с источниками.
10. Прогнать бриф через quality gate.

Если pipeline не может заполнить один из обязательных разделов, он должен честно пометить пробел, а не генерировать уверенный, но слабый текст.
