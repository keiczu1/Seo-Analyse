# Спецификация финального ТЗ-брифа страницы

Моя версия документа, описывающего идеальный результат SEO-анализа: финальный ТЗ-бриф на страницу. Основана на `ideal-final-page-brief.md` и `final-page-brief-spec.md` с дополнениями: таблица автоматизации, fallback-сценарии, worked example SEO-триплетов, guidance по объёму, связь триплетов со schema, единая нумерация.

## Главный принцип

Финальный бриф — это не SEO-отчёт, не выгрузка конкурентов, не список ключей, не outline и не черновик статьи. Это **рабочее задание на страницу**.

После чтения брифа должно быть понятно:

- какую страницу нужно создать;
- зачем она нужна пользователю;
- какой intent она закрывает;
- какие запросы входят в страницу, а какие нет;
- что именно должен написать копирайтер;
- какие блоки и элементы нужны на странице;
- какие факты нужно подтвердить источниками;
- чем страница должна быть лучше конкурентов;
- как принять готовый текст и готовую страницу.

Формат документа универсальный. Структура страницы и содержание всегда зависят от анализа.

---

## 1. Короткое решение по странице

Первый блок брифа. За 30 секунд объясняет суть.

Обязательные поля:

- основной запрос;
- тип страницы;
- решение: создать новую, обновить существующую, объединить, разделить или не делать;
- главная задача страницы;
- главный риск;
- главный способ выиграть у конкурентов;
- статус готовности: `ready_for_copywriter`, `ready_for_design_and_build`, `requires_expert_review`, `blocked_until_sources_refreshed`, `draft`.

Формат:

```markdown
## Короткое решение

Делаем: <тип страницы>
Почему: <логика intent и SERP>
Для кого: <аудитория>
Главная задача: <что пользователь должен решить>
Главное отличие: <чем будем лучше среднего конкурента>
Риск: <что может испортить страницу>
Статус: <...>
```

---

## 2. Паспорт страницы

Справочный блок. Не превращать в рассуждение.

Обязательные поля:

- основной запрос;
- рабочее название страницы;
- язык и регион;
- дата анализа;
- тип страницы;
- поисковый intent;
- этап воронки;
- аудитория;
- уровень знаний пользователя;
- YMYL-риск;
- нужна ли экспертная проверка;
- существующие страницы, с которыми возможна cannibalization;
- целевое действие пользователя.

---

## 3. Intent и пользовательская задача

Не «о чём запрос», а «что человек пытается сделать».

Обязательные вопросы:

- Что пользователь хочет получить сразу?
- Какую проблему он решает?
- Какой результат он считает успешным?
- Что он уже, вероятно, знает?
- Какие страхи, сомнения или ограничения у него есть?
- Насколько запрос информационный, коммерческий, транзакционный, навигационный или смешанный?
- Какие intent нельзя смешивать на этой странице?

Пример хорошей формулировки:

```markdown
Пользователь не просто ищет определение. Ему нужен практический порядок действий,
развилка по ситуациям, список документов, ограничения, сроки и понимание рисков.
Поэтому страница должна начинаться с короткого ответа и сценариев, а не с общего вступления.
```

## 4. Кластер запросов и границы страницы

Кластер нужен не для вставки ключей, а для проектирования страницы.

### Группы кластера

- **Core** — запросы, которые страница обязана закрыть напрямую.
- **Supporting** — вопросы, которые раскрываются внутри разделов.
- **Adjacent** — близкие темы для краткого упоминания или внутренней ссылки.
- **Excluded** — запросы с другим intent, которые нельзя покрывать здесь.

### Правила

- дубли и морфологические варианты объединяются;
- разные формулировки одного смысла не раздувают outline;
- каждый supporting-запрос привязан к конкретному разделу;
- excluded-запросы обязательны, чтобы не получить размытую страницу;
- если нет внешних сервисов частотности, нельзя писать «высокочастотный» или «низкочастотный» без данных — пометить `synthetic/serp-derived`.

Формат:

```markdown
### Core
- <запрос> -> закрываем в первом экране и разделе <...>

### Supporting
- <вопрос> -> раздел <...>

### Adjacent
- <тема> -> коротко упомянуть / дать ссылку / вынести отдельно

### Excluded
- <запрос> -> не покрываем, потому что <причина>
```

## 5. Сущности, обязательные факты и SEO-триплеты

Этот раздел описывает предметную область: какие сущности есть в теме, какие факты пользователь обязан получить, какие связи между ними нужно объяснить и чем можно обогатить страницу сверх пересказа SERP.

### 5.1. Определения

- **Сущность** — главный объект темы: услуга, процедура, инструмент, болезнь, товар, закон, показатель, место, событие, человек, организация.
- **Обязательный факт** — информация, без которой ответ будет неполным или опасным.
- **SEO-триплет** — смысловая единица вида `сущность -> свойство/отношение -> значение`, иногда с условиями, датой, источником и ограничениями.

### 5.2. Обязательные факты

Обязательный факт отвечает хотя бы одному критерию:

- без него пользователь не может действовать;
- без него ответ будет вводить в заблуждение;
- его ожидает выдача и пользователь;
- он влияет на безопасность, деньги, здоровье, юридический результат или решение о покупке;
- он часто путается в конкурентных материалах;
- он является основанием для выбора сценария.

Формат:

```markdown
| Факт | Почему обязателен | Где раскрыть | Источник | Условия/исключения | Риск ошибки |
|---|---|---|---|---|---|
```

### 5.3. SEO-триплеты

Базовый формат:

```
<сущность> -> <свойство или отношение> -> <значение>
```

Расширенный формат:

```
Сущность:
Отношение:
Значение:
Условие:
Источник:
Дата актуальности:
Где раскрыть:
Тип: baseline | enrichment | risk | excluded
```

Типы триплетов:

- **baseline** — обязательный минимум, без которого страница неполная;
- **enrichment** — обогащение, которое делает страницу полезнее конкурентов;
- **risk** — утверждение, где нужна осторожность, источник или экспертная проверка;
- **excluded** — смысл, который относится к другой странице и не должен размывать intent.

### 5.4. Полный worked example: «ипотечный калькулятор» (G01)

```markdown
Главная сущность: ипотечный кредит (как финансовый продукт)
Связанные сущности: ежемесячный платёж, процентная ставка, первоначальный взнос,
срок кредита, тип платежа, переплата, график платежей, рефинансирование.

### SEO-триплеты

| Сущность | Отношение | Значение | Тип | Где использовать | Проверка |
|---|---|---|---|---|---|
| ипотечный кредит | имеет параметр | процентная ставка | baseline | первый экран, форма калькулятора | ставка зависит от банка и даты |
| ипотечный кредит | имеет параметр | первоначальный взнос | baseline | первый экран, форма калькулятора | мин. взнос зависит от программы |
| ипотечный кредит | имеет тип платежа | аннуитетный / дифференцированный | baseline | раздел «Типы платежей» | объяснить разницу простыми словами |
| ежемесячный платёж | формула расчёта | A = P × [r(1+r)^n] / [(1+r)^n - 1] | enrichment | раздел «Как считается» | проверить формулу у финансиста |
| переплата | вычисляется как | сумма всех платежей − тело кредита | enrichment | раздел результатов | показать отдельно от тела кредита |
| график платежей | формат вывода | таблица: месяц, платёж, основной долг, проценты, остаток | enrichment | раздел «График платежей» | таблица должна быть crawlable HTML |
| результат калькулятора | является | предварительным, не решением банка | risk | дисклеймер на первом экране | обязателен для YMYL |
| рефинансирование | можно рассмотреть при | снижении ставки на 2+ п.п. | enrichment | раздел «Что влияет на переплату» | не гарантировать выгоду |
| ипотечный кредит в другой стране | excluded | другое законодательство и ставки | excluded | не включать | отдельная страница |
| одобрение заявки | excluded | зависит от банка и кредитной истории | excluded | не включать | отдельная страница |

### Обогащение
- Показать чувствительность платежа к изменению ставки на ±1-2 п.п.
- Выводить график платежей таблицей, а не только итоговую сумму.
- Отдельно показывать переплату, тело кредита и проценты.
- Добавить сравнение аннуитетного и дифференцированного платежа на одном примере.

### Не включать
- Одобрение заявки -> другой intent (подача, а не расчёт).
- Ипотека в другой стране -> другое законодательство.
- Страхование ипотеки -> отдельная тема, упомянуть ссылкой.
```

### 5.5. Как использовать триплеты

Триплеты должны попадать не отдельной сырой таблицей, а в решения:

- в первый экран, если это главный ответ;
- в outline, если это обязательный раздел;
- в таблицу фактов, если нужна проверка источником;
- в UX-блок, если связь лучше показать таблицей, схемой или шагами;
- в FAQ, если это реальный следующий вопрос пользователя;
- в excluded-темы, если смысл нельзя покрывать на этой странице;
- в schema, если триплет поддерживает видимую структурированную информацию.

### 5.6. Связь триплетов со schema

| Тип триплета | Schema | Правило |
|---|---|---|
| baseline с чёткими значениями | поддерживает Article, HowTo, FAQPage | размечать только если значение видимо на странице |
| baseline с числовыми данными | поддерживает Product, Offer, QuantitativeValue | данные должны быть актуальны и видимы |
| enrichment | обычно не размечается | это добавочная ценность текста, не структурированных данных |
| risk | **НЕ размечается** без экспертной проверки | спорные утверждения в schema — нарушение правил |
| excluded | **НЕ попадает** в schema этой страницы | относится к другой странице |

### 5.7. Как триплеты помогают не делать rewrite

Конкуренты показывают, что уже есть в выдаче. Сущности и триплеты показывают, что тема реально требует раскрыть.

Бриф должен сравнивать:

- что обязано быть по сущности темы;
- что уже есть у конкурентов;
- чего у конкурентов нет;
- что можно объяснить понятнее;
- какие связи лучше показать структурно;
- какие факты нужно проверить у первоисточников.

Принцип: не «у конкурентов есть раздел X, добавим раздел X», а «пользователю нужно понять связь A -> B -> C, конкуренты раскрывают её слабо, поэтому мы делаем таблицу/схему/пример».

Формат итогового блока:

```markdown
## Сущности, факты и SEO-триплеты

Главная сущность:
Связанные сущности:

### Обязательные факты
| Факт | Почему обязателен | Где раскрыть | Источник | Риск |
|---|---|---|---|---|

### SEO-триплеты
| Сущность | Отношение | Значение | Тип | Где использовать | Проверка |
|---|---|---|---|---|---|

### Обогащение
- <какую связь/пример/таблицу/сценарий добавить>

### Не включать
- <сущность или связь> -> <почему это другой intent>
```

## 6. Конкурентная картина

Цель — не собрать «что есть у конкурентов», а понять, какие решения должна принять наша страница.

### 6.1. Кого считать конкурентами

- **Поисковые конкуренты** — страницы в SERP по запросу. Показывают, что поисковая система считает релевантным.
- **Контентные конкуренты** — страницы, которые лучше закрывают задачу, даже если не в топе по основному запросу.
- **Бизнес-конкуренты** — сайты, с которыми проект конкурирует за заявку, продажу или внимание.

### 6.2. Сколько конкурентов анализировать

- Минимум: топ-5 органических результатов + 2-3 по supporting-запросам.
- Для YMYL: обязательно добавить официальные источники.
- Не нужно анализировать всё подряд. Лучше 8 хорошо разобранных страниц, чем 30 URL в таблице без выводов.

### 6.3. Что фиксировать по каждому конкуренту

Минимальная карточка (сырьё для анализа, не попадает в бриф целиком):

```markdown
URL:
Тип сайта / Тип страницы:
Позиция / источник:
Title / H1 / Snippet:
Дата обновления / Автор:
Что видно на первом экране:
Сильные блоки / Слабые места:
Источники и доказательства:
UX-решения / CTA:
Что можно использовать как сигнал:
Что нельзя копировать:
```

### 6.4. Цепочка «Сигнал -> Решение»

Каждый конкурентный сигнал проходит цепочку:

```
Сигнал -> Значение -> Решение для нашей страницы -> Где в ТЗ -> Критерий приёмки -> Что не копировать
```

Пример:

```markdown
Сигнал: 4 из 5 конкурентов дают список документов, но прячут его после длинного вступления.
Значение: список документов — обязательное ожидание, но текущая выдача неудобна.
Решение: дать короткий список документов в первом экране и подробную таблицу ниже.
Где в ТЗ: первый экран, раздел «Документы», UX-блок «чек-лист».
Критерий приёмки: пользователь видит базовый список до скролла.
Что не копировать: длинное вступление перед практической информацией.
```

Если сигнал не приводит к решению, его не нужно включать в финальный бриф.

### 6.5. Baseline и преимущество

- **Baseline** — то, без чего страница не будет соответствовать выдаче и ожиданиям.
- **Преимущество** — то, что делает страницу лучше конкурентов.

Нельзя продавать baseline как уникальное преимущество.

### 6.6. Типовые ошибки

Нельзя:

- копировать структуру лидера без понимания intent;
- считать частоту H2 доказательством важности;
- переносить все разделы конкурентов в нашу страницу;
- смешивать поисковых и бизнес-конкурентов без пометки;
- делать выводы по одному конкуренту;
- путать «конкуренты делают» и «пользователю нужно».

### 6.7. Формат в финальном брифе

```markdown
## Конкурентная картина

### Доминирующий тип выдачи
<какие страницы ранжируются и что это значит>

### Baseline-ожидания
- <элемент> -> <почему обязателен>

### Пробелы конкурентов
- <проблема> -> <как используем>

### Решения для нашей страницы
| Сигнал | Что значит | Решение | Где в ТЗ | Приёмка | Не копировать |
|---|---|---|---|---|---|
```

## 7. Стратегия нашей страницы

Обязательные поля:

- какую страницу делаем;
- какой формат выбираем;
- что будет главным ответом;
- как пользователь будет двигаться по странице;
- что является обязательным минимумом;
- где будет наше преимущество;
- что не включаем;
- какие риски нужно контролировать.

Формат:

```markdown
Стратегия:
Мы делаем <тип страницы>, которая сначала даёт <быстрый ответ>,
затем помогает выбрать <сценарий>, потом раскрывает <важные детали>
и завершает <следующим шагом>. Страница выигрывает не объёмом текста,
а <конкретное преимущество>.
```

---

## 8. Title, H1, description и первый экран

Требования к первому контакту пользователя со страницей.

Обязательные поля:

- основной `title`;
- альтернативный `title`;
- `H1`;
- `meta description`;
- первый абзац;
- блок «коротко», если нужен;
- сценарные кнопки или якоря, если нужны;
- дата обновления, автор, эксперт или источник доверия;
- первый CTA, если уместен.

Требования:

- `title` не обещает больше, чем есть на странице;
- `H1` понятен человеку, а не только поисковой системе;
- первый экран подтверждает intent;
- для YMYL — сразу показать актуальность и доверие.

## 9. Подробный план контента для копирайтера

Центральная часть брифа. Для каждого раздела:

```markdown
### H2: <заголовок>

Задача раздела:
Вопрос пользователя:
Что написать:
Обязательные факты:
Источники:
Формат блока: текст | таблица | список | схема | чек-лист | пример | форма | калькулятор
Покрываемые запросы:
Не писать:
Критерий готовности:
```

Раздел нельзя включать, если у него нет задачи. Outline — карта смысла, а не список заголовков.

## 10. UX, визуальная структура и функциональные блоки

Страница — не только текст. Бриф описывает, как пользователь воспринимает информацию.

### 10.1. Принцип выбора блоков

```
Если пользователь должен сравнить варианты — нужна таблица или карточки.
Если пользователь должен выполнить процесс — нужны шаги.
Если пользователь должен проверить готовность — нужен чек-лист.
Если пользователь должен выбрать сценарий — нужна развилка.
Если пользователь должен рассчитать результат — нужен калькулятор.
```

### 10.2. Обязательные поля

- композиция первого экрана;
- порядок основных блоков;
- нужна ли якорная навигация;
- где нужны таблицы, чек-листы, карточки, предупреждения, интерактивные элементы;
- как блоки работают на мобильном;
- какие элементы доверия должны быть видны.

### 10.3. Визуальные требования

Визуальная часть помогает понять материал, а не украшает страницу.

- Для сложных тем предпочтительны структурные элементы: таблицы, схемы, шаги, чек-листы, сравнения.
- Декоративные изображения не заменяют объяснение.
- Стиль соответствует теме: спокойный справочный, юридически строгий, коммерческий, продуктовый, экспертный.
- Предупреждения выделяются визуально.

---

## 11. Источники, факты и экспертная проверка

Обязателен для YMYL-тем: право, медицина, финансы, налоги, госуслуги, безопасность.

Обязательные поля:

- какие факты требуют подтверждения;
- какие источники приоритетны;
- какие данные могут устареть;
- что перепроверить перед публикацией;
- где нужна экспертная вычитка;
- какие утверждения запрещены;
- какие формулировки должны быть осторожными.

Формат:

```markdown
| Факт | Источник | Где используется | Проверить перед публикацией | Риск |
|---|---|---|---|---|
```

Нельзя:

- придумывать сроки, ставки, цены, законы, документы;
- ссылаться на конкурента как на первоисточник;
- писать «гарантированно», если результата нельзя гарантировать;
- скрывать дату актуальности в темах, где она важна.

---

## 12. Внутренние ссылки и следующий шаг пользователя

Обязательные поля:

- какие темы вынести на отдельные страницы;
- куда ссылаться внутри текста;
- какой следующий шаг логичен;
- где уместен CTA;
- какие ссылки помогают пользователю, а не только SEO.

Для информационных запросов CTA аккуратный. Для коммерческих — полезный ответ не исчезает за продажей.

---

## 13. Schema и технические SEO-требования

Schema добавляется только под видимый контент.

Обязательные поля:

- рекомендуемый тип разметки;
- какие видимые блоки её поддерживают;
- `canonical`;
- `index/noindex`;
- `breadcrumbs`;
- требования к H-структуре;
- требования к датам;
- технические ограничения.

Запрещено:

- FAQ-разметка без видимого FAQ;
- HowTo без реальных шагов;
- AggregateRating без реальных рейтингов;
- разметка сущностей, которых нет на странице.

Связь с SEO-триплетами: см. раздел 5.6.

---

## 14. Редакционные правила

Писать:

- прямо и по делу;
- конкретно;
- с практической пользой;
- с примерами, где без них непонятно;
- с объяснением терминов;
- с отделением фактов от рекомендаций;
- в тоне, подходящем теме.

Не писать:

- длинные вступления;
- SEO-воду;
- повторы одного смысла;
- общие фразы без решения;
- искусственные ключевые фразы;
- неподтверждённые утверждения;
- кликбейт;
- имитацию экспертности.

```markdown

- [ ] Supporting-запросы распределены по разделам.

- [ ] Baseline и преимущества разделены.
- [ ] Каждый раздел outline имеет задачу и критерий готовности.
- [ ] Факты и источники сопоставлены с разделами.
- [ ] Для YMYL-тем указана экспертная проверка.
- [ ] Первый экран описан как UX-блок, а не только как текст.
- [ ] Таблицы, чек-листы, формы и интерактивы указаны там, где реально нужны.
- [ ] Визуальные элементы помогают решению задачи, а не украшают.
- [ ] Внутренние ссылки помогают пользовательскому пути.
```

## 16. Fallback-сценарии: что делать, если данные неполные

Бриф должен честно показывать, каких данных не хватает, а не генерировать уверенный, но слабый текст.

| Сценарий | Статус | Что делать |
|---|---|---|
| SERP API недоступен (down, квота) | `draft` | Бриф строится только на анализе запроса. Пометка «без конкурентной картины». Разделы 6, 8, 10 заполняются как предположения с пометкой `unverified`. |
| Все конкуренты заблокированы (blocked) | `draft` или `requires_expert_review` | Анализ по title/snippet из SERP без парсинга. Пометка «ограниченный анализ, только SERP-мета». |
| Запрос без SERP-истории (новый, уникальный) | `draft` | Только query-only классификация. Пометка `no_serp_data`. |
| Кластер недоступен (нет внешних данных) | Любой, кроме `ready_for_design_and_build` | Используется только primary query. Пометка `single_query_no_cluster`. |
| YMYL-тема без подтверждённых источников | `requires_expert_review` | Бриф может быть заполнен, но статус не повышается до `ready_for_copywriter` без экспертной проверки. |
| Часть разделов не может быть заполнена | `draft` | Незаполненные разделы помечены `data_unavailable`, а не удалены. Pipeline честно указывает пробел. |

Правило: если pipeline не может заполнить один из обязательных разделов, он **помечает пробел**, а не генерирует уверенный, но слабый текст.

## 17. Guidance по объёму брифа

Объём зависит от типа запроса и сложности темы. Ориентиры:

| Тип запроса | Архетип | Ожидаемый объём | Примечания |
|---|---|---|---|
| reference (справочный) | `reference_database` | 2-3 страницы | Короткий ответ + таблица + источники |
| freshness (актуальность) | `freshness_page` | 2-3 страницы | Статус + хронология + источники |
| local (локальная услуга) | `local_service_page` | 3-4 страницы | Услуга + условия + доверие + контакт |
| how-to (инструкция) | `how_to_guide` | 4-6 страниц | Шаги + условия + риски + проверка |
| definition (определение) | `article_guide_definition` | 4-6 страниц | Определение + контекст + примеры + связанные понятия |
| comparison (сравнение) | `comparison_page` | 5-8 страниц | Таблица + сценарии + методология |
| hybrid tool + guide | `hybrid_tool_guide` | 4-7 страниц | Инструмент + методика + примеры + ограничения |
| commercial (коммерческое сравнение) | `commercial_comparison_page` | 5-8 страниц | Shortlist + критерии + цены + методология + disclosure |
| YMYL guide | `ymyl_medical_guide` | 5-8 страниц | Осторожный ответ + источники + expert review |
| hub (неоднозначность) | `hub_or_disambiguation` | 2-3 страницы | Развилка по смыслам + ссылки на подстраницы |

Правило: бриф для запроса «калории в авокадо» не должен быть 15 страниц. Бриф для «как открыть ИП» не должен быть 2 страницы. Объём определяется задачей, а не желанием наполнить.

## Как этот документ влияет на pipeline

Pipeline строится не вокруг того, что легко собрать, а вокруг того, что требуется финальному брифу.

Будущий workflow:

1. Определить intent и тип страницы.
2. Собрать и нормализовать кластер.
3. Выбрать релевантных конкурентов.
4. Разобрать конкурентов по intent, первому экрану, блокам, глубине, доверию, UX и пробелам.
5. Сгенерировать SEO-триплеты и обязательные факты.
6. Превратить конкурентные сигналы в решения.
7. Разделить baseline и преимущества.
8. Собрать section-by-section ТЗ для копирайтера.
9. Добавить UX и визуальные требования.
10. Сопоставить факты с источниками.
11. Связать триплеты с schema.
12. Прогнать бриф через quality gate.
13. Если данные неполные — честно пометить пробел (раздел 16).

Если pipeline не может заполнить один из обязательных разделов, он помечает пробел, а не генерирует уверенный, но слабый текст.
