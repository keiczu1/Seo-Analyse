# Эталонный итоговый ТЗ-бриф для страницы

Этот документ описывает, каким должен быть финальный артефакт SEO-анализа: не отчет, не черновик статьи и не HTML-макет, а рабочее техническое задание на страницу. По нему копирайтер должен написать сильный контент, редактор проверить полноту и точность, а разработчик понять, какие блоки и интерфейс нужны на странице.

Главная идея: у нас один универсальный формат ТЗ-брифа, но не один универсальный outline. Каркас документа одинаковый для любого запроса, а структура страницы, глубина, блоки, источники и сценарии меняются по результатам анализа интента, SERP, кластера и требований темы.

## Назначение

Финальный ТЗ-бриф должен отвечать на четыре вопроса:

1. Какую страницу нужно создать под запрос и почему именно такую.
2. Какой пользовательский сценарий страница должна закрыть.
3. Что именно должен написать копирайтер, по разделам, фактам и смысловым задачам.
4. Как страница должна быть устроена визуально и функционально, чтобы контент был удобен, убедителен и конкурентоспособен.

Хороший бриф не просит "написать статью про тему". Он дает конкретную задачу: что пользователь хочет решить, какие вопросы нужно закрыть, где нужны доказательства, какие блоки обязательны, что нельзя утверждать без источника и как проверить готовый результат.

## Пользователи документа

- **Копирайтер**: пишет основной текст, заголовки, пояснения, примеры, FAQ и микро-копирайтинг для блоков.
- **Редактор**: проверяет полноту, отсутствие воды, фактическую точность, тон, структуру и соответствие задаче.
- **SEO-специалист**: проверяет интент, кластер, покрытие SERP, title/H1, внутренние ссылки, schema и каннибализацию.
- **Разработчик или дизайнер**: понимает, какие блоки, таблицы, формы, интерактивные элементы и состояния нужны на странице.

## Входные данные

Перед созданием ТЗ-брифа workflow должен собрать и нормализовать:

- основной запрос;
- регион, язык, тип поисковой системы и дату анализа;
- SERP и конкурентов, если доступны;
- поисковый интент и возможные смешанные интенты;
- кластер запросов: основные, поддерживающие, соседние и исключенные;
- источники фактов: официальные, экспертные, редакционные, внутренние;
- ограничения проекта: тип сайта, бренд, юридические риски, коммерческая цель, существующие страницы;
- решение по странице: создать новую, обновить существующую, объединить, разделить, не делать страницу.

Если часть данных недоступна, бриф должен явно пометить это. Нельзя маскировать отсутствие анализа уверенными формулировками.

## Статус готовности

У каждого финального брифа должен быть один статус:

- `ready_for_copywriter` - можно писать контент.
- `ready_for_design_and_build` - можно писать контент и проектировать страницу.
- `requires_expert_review` - тема требует экспертной проверки до публикации.
- `blocked_until_sources_refreshed` - ключевые факты устарели или не подтверждены.
- `draft` - бриф неполный и не должен идти в работу.

Статус ставится в начале документа вместе с коротким объяснением.

## Обязательная структура финального ТЗ-брифа

### 1. Паспорт страницы

В этом блоке фиксируется решение по странице:

- основной запрос;
- рабочее название страницы;
- тип страницы: статья, инструкция, сервисная страница, обзор, сравнение, калькулятор, каталог, лендинг, справочник, FAQ-страница или гибрид;
- поисковый интент;
- целевое действие пользователя;
- аудитория;
- регион и дата актуальности;
- статус готовности;
- нужна ли экспертная проверка.

Паспорт должен быть коротким. Это не место для длинного анализа.

### 2. Решение по интенту

Нужно объяснить, какую задачу пользователя закрываем.

Обязательные поля:

- что пользователь хочет получить сразу;
- какую проблему он решает;
- насколько запрос информационный, коммерческий, навигационный, транзакционный или смешанный;
- какие интенты не покрываем на этой странице;
- что должно быть видно на первом экране.

Пример формата:

```markdown
Пользователь хочет понять, что делать в ситуации X, какие есть варианты, какие документы/шаги/условия нужны, сколько это займет и где есть риски. Первый экран должен сразу дать короткий ответ и путь выбора, а не начинаться с общего вступления.
```

### 3. Кластер запросов

Кластер нужен не для набивки ключей, а для понимания границ страницы.

В брифе должны быть четыре группы:

- **Core**: запросы, которые страница обязана закрыть напрямую.
- **Supporting**: вопросы и формулировки, которые раскрываются внутри разделов.
- **Adjacent**: близкие темы, которые можно кратко упомянуть и связать внутренними ссылками.
- **Excluded**: запросы, которые нельзя смешивать с этой страницей, потому что у них другой интент или нужна отдельная страница.

Правила:

- дубли нормализуются: разные формы одного запроса не должны повторяться как отдельные смыслы;
- сырые H2 конкурентов не считаются ключами сами по себе;
- если кластер собран без внешних сервисов, это нужно пометить как `synthetic/serp-derived`, без заявлений о частотности;
- каждый важный supporting-запрос должен быть привязан к разделу outline.

### 4. Конкурентная картина

Этот блок не должен быть пересказом конкурентов. Он должен объяснять, что их выдача означает для нашей страницы.

Нужно зафиксировать:

- какие типы страниц доминируют в SERP;
- какие блоки встречаются чаще всего;
- какие вопросы конкуренты закрывают хорошо;
- какие пробелы у конкурентов можно использовать;
- где выдача требует осторожности: YMYL, официальные источники, свежесть данных, экспертность;
- что мы должны сделать лучше, чем средний конкурент.

Формат должен быть decision-first:

```markdown
Сигнал SERP: конкуренты начинают с общего определения.
Вывод для нашей страницы: первый экран должен сразу дать практический ответ и развилку по сценариям, потому что пользователь уже знает тему и ищет решение.
```

### 5. Контентная стратегия страницы

Этот блок задает смысловую линию будущего текста.

Обязательные элементы:

- главная мысль страницы в одном абзаце;
- обещание страницы пользователю;
- уровень глубины: базовый, подробный, экспертный, практический;
- тон: спокойный, деловой, юридически аккуратный, экспертный, простой, коммерческий и т.п.;
- что должно быть доказано источниками;
- какие утверждения запрещены;
- какие примеры, таблицы, чек-листы или сценарии нужны;
- где нужны предупреждения или дисклеймеры.

### 6. Title, H1, Description

Бриф должен давать не один случайный вариант, а рабочий набор:

- основной `title`;
- альтернативный `title`, если нужен более мягкий или более коммерческий вариант;
- `H1`;
- `meta description`;
- правило для первого абзаца.

Требования:

- `title` и `H1` не должны быть механическими дублями, если это ухудшает читаемость;
- основной запрос должен быть естественно отражен в `title`, `H1` или первом экране;
- нельзя обещать больше, чем страница реально закрывает;
- для YMYL-тем нельзя использовать кликбейт и гарантии результата.

### 7. Первый экран

Первый экран должен закрывать ожидание пользователя до скролла.

В брифе нужно указать:

- что пользователь видит первым;
- какой короткий ответ или выбор получает;
- нужна ли навигация по сценариям;
- нужен ли блок "коротко";
- какие CTA допустимы;
- какие элементы доверия нужны: дата обновления, автор, эксперт, источники, предупреждение.

Принцип: первый экран не должен быть длинным вступлением ради SEO. Он должен подтвердить пользователю, что страница решает его задачу.

### 8. Outline с задачами разделов

Outline должен быть не списком заголовков, а рабочей картой текста.

Для каждого раздела нужно указать:

- рекомендуемый заголовок;
- цель раздела;
- какой пользовательский вопрос закрывается;
- какие факты или источники нужны;
- какие supporting-запросы покрываются;
- какой формат нужен: текст, таблица, список, схема, чек-лист, пример, форма, калькулятор;
- что нельзя писать в этом разделе;
- критерий готовности.

Формат:

```markdown
## <H2>

Задача раздела:
Вопрос пользователя:
Что написать:
Факты и источники:
Формат блока:
Покрываемые запросы:
Не писать:
Критерий готовности:
```

Если раздел не имеет понятной задачи, его нельзя включать в outline.

### 9. Требования к фактам и источникам

Особенно важно для юридических, финансовых, медицинских, государственных и других YMYL-тем.

Бриф должен содержать:

- список фактов, которые требуют подтверждения;
- рекомендуемые источники;
- дату, на которую данные должны быть актуальны;
- что нужно перепроверить перед публикацией;
- какие формулировки использовать осторожно;
- где нужна экспертная вычитка.

Нельзя придумывать факты, сроки, цены, законы, ставки, процедуры или требования без источников.

### 10. UX и структура страницы

Бриф должен объяснять не только текст, но и страницу как пользовательский интерфейс.

Нужно указать:

- общую композицию страницы;
- какие блоки должны быть выше основного текста;
- нужна ли боковая навигация или якорное меню;
- где нужны таблицы;
- где нужны чек-листы;
- где нужны сценарные развилки;
- где нужны формы, калькуляторы или интерактивные элементы;
- как страница должна работать на мобильном;
- какие состояния нужны для интерактивных блоков;
- какие элементы доверия должны быть видны.

Для информационной страницы нельзя автоматически делать маркетинговый лендинг. Для сервисной или коммерческой страницы нельзя прятать действие пользователя в конце длинной статьи.

### 11. Визуальные требования

Визуальная часть должна помогать понять материал, а не украшать страницу.

Бриф должен определить:

- нужна ли иллюстрация, фото, схема, таблица, карточки, timeline, stepper, comparison-block;
- какие визуальные элементы обязательны;
- какие визуальные элементы нежелательны;
- как выделять важные предупреждения;
- какой стиль подходит теме: спокойный справочный, юридически строгий, коммерческий, продуктовый, экспертный и т.п.

Для сложных тем предпочтительны структурные визуальные элементы: таблицы, схемы, шаги, чек-листы, сравнения. Декоративные изображения не должны заменять объяснение.

### 12. Внутренние ссылки и дальнейший путь

Бриф должен указать:

- куда пользователь должен перейти после чтения;
- какие страницы нужно связать внутренними ссылками;
- какие темы лучше вынести в отдельные страницы;
- какие ссылки нужны в supporting-разделах;
- какие CTA допустимы и где они должны стоять.

Внутренние ссылки должны помогать сценарию пользователя, а не существовать только ради SEO.

### 13. Schema и технические SEO-требования

Этот блок нужен только если разметка действительно соответствует видимому содержанию.

В брифе можно указать:

- рекомендуемый тип schema;
- какие поля можно заполнить;
- какие FAQ можно размечать;
- какие хлебные крошки нужны;
- canonical;
- robots/indexing-решение;
- требования к заголовкам;
- требования к обновлению даты.

Запрещено добавлять FAQ schema, если вопросов нет на видимой странице, или размечать сущность, которой фактически нет.

### 14. Редакционные требования

Копирайтер должен получить понятные правила качества:

- писать прямо и по делу;
- не начинать с банальных вступлений;
- не повторять один и тот же смысл разными словами;
- не использовать SEO-воду;
- не имитировать экспертность;
- объяснять термины там, где они мешают пониманию;
- приводить примеры там, где без них пользователь не сможет применить совет;
- отделять факты от рекомендаций;
- не обещать гарантированный результат.

Текст должен быть полезным даже без знания SEO-анализа.

### 15. Приемка готового контента

В конце брифа должен быть чек-лист приемки:

- первый экран отвечает на основной запрос;
- структура страницы соответствует интенту;
- каждый H2 имеет понятную задачу;
- core-запросы закрыты напрямую;
- supporting-запросы распределены по разделам;
- excluded-запросы не смешаны с темой;
- факты подтверждены источниками;
- нет воды и общих рассуждений;
- есть практический путь для пользователя;
- страница понятна на мобильном;
- визуальные блоки помогают решению задачи;
- CTA и внутренние ссылки уместны;
- schema соответствует видимому содержанию;
- бриф достаточен для работы без дополнительных устных пояснений.

## Антипаттерны

Финальный бриф считается плохим, если в нем есть:

- список H2 без объяснения задач разделов;
- общий текст "написать полезную статью";
- механический список ключей без кластеризации;
- дубли ключей как отдельные смыслы;
- пересказ конкурентов без выводов;
- требования "добавить FAQ", если непонятно зачем;
- универсальный outline для всех запросов;
- рекомендации без источников в YMYL-темах;
- HTML вместо контентного ТЗ;
- статья вместо ТЗ;
- дизайн без понимания текста;
- SEO-отчет, из которого нельзя понять, что писать.

## Критерий идеального результата

Идеальный финальный ТЗ-бриф - это документ, после которого не нужно спрашивать "о чем писать", "какие разделы нужны", "что важно для SEO", "какие блоки делать" и "как понять, что страница готова". Он превращает анализ запроса в конкретное задание на страницу: содержание, структуру, доказательства, интерфейс, ограничения и приемку.
